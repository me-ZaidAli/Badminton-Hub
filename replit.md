# Club Master

Club Master is a comprehensive web application for racket sports clubs, streamlining operations and enhancing player engagement.

## Run & Operate

_Populate as you build_

## Stack

- **Frontend**: React 18, TypeScript, Wouter, TanStack React Query, Tailwind CSS, shadcn/ui
- **Backend**: Node.js, Express.js, TypeScript
- **Database**: PostgreSQL with Drizzle ORM
- **Build Tools**: Vite (frontend), esbuild (backend)
- **Other**: express-session, passport/passport-local, date-fns, recharts, framer-motion

## Where things live

- `client/`: Frontend source code.
  - `client/src/index.css`: Contains base CSS tokens for "Elite Sports" theme.
  - `client/src/hooks/use-theme.ts`: Manages theme application classes.
- `server/`: Backend source code.
- `drizzle/schema.ts`: Database schema definition.

## Architecture decisions

- **Theming**: Advanced Premium Theme System with 64 themes, optimized for various displays including AMOLED, featuring a global Ultra-Premium Transparent Glass UI. Default palette ("Dribbble Blue-Violet") in `client/src/index.css` `:root`/`.dark`: deep blue-violet background (232 40% 10%), lifted blue-violet glass cards (232 32% 16%), violet accent (252 90% 68%), white foreground; body radial wash on `html.theme-default` matches the reference screenshot (brighter violet/blue from top, deepening at bottom).
- **Matchmaking**: Deterministic Smart Match Engine v6 with multi-mode, gender-aware, 9-tier grade-based scoring, complemented by BPG Competitive Balance Engine and Session Fairness Command Center.
- **Freemium Model**: Backend-enforced and frontend-gated 2-plan (Basic FREE, Premium) freemium structure.
- **Multi-Tenancy**: Supports multi-club and multi-sport management with granular, role-based access control.
- **Club Control Center** (`/admin/control-center`): Single consolidated hub (page: `client/src/pages/admin/ClubControlCenter.tsx`) replacing scattered Club Management / Billing / All Clubs tiles. OWNER sees all clubs (via `/api/super-admin/clubs/billing`), ADMIN/club-admin sees their own admin clubs. Per-club feature toggles persisted in `clubs.featureOverrides` (jsonb) and exposed via `GET/PATCH /api/clubs/:id/feature-overrides` (PATCH = OWNER-only).
- **Localization**: All event date/time inputs and displays are anchored to `Europe/London` (BST/GMT auto-aware).
- **BSL Player Profile Dashboard**: `/bsl/profile` consumes a single `GET /api/bsl/players/me/dashboard` round-trip in `server/bsl-routes.ts` (~L1158) returning `{player, club, league:{name,venueName,nextLeagueDay,categoryFees,playerFee}, categories:[{category,team,partners[]}], nextMatch, upcoming[], history[], walletTx[]}`. Partners resolve via `bslTeamMembers→bslPlayers→users` (excludes self, hydrates `displayName`/`fullName`/`profilePictureUrl`). Match history + next match resolve via `bslRubbers→bslFixtures→bslTeams→bslClubs`; **per-fixture home/away perspective is derived from the player's actual rubber slot** (`homePlayer1/2Id` vs `awayPlayer1/2Id`), with team-membership / `bslPlayers.bslTeamId` only used as a fallback when the player isn't yet slotted into any rubber for that fixture — this prevents WIN/LOSS inversion when membership rows are stale. Outcome (`WIN`/`LOSS`/`DRAW`) is computed from `homeRubbers` vs `awayRubbers` via the resolved side. UI in `client/src/pages/bsl/PlayerProfile.tsx` renders hero (club logo + division + status + W/L/Win% stat strip), per-category pair cards with partner avatar + name, next-match card with team logos + court + countdown, and last-20 match history with WIN/LOSS pills. Registration error toast strips the `"402: "` prefix from `apiRequest` errors and uses neutral `"Heads up"` title (instead of alarming `"Cannot register"`) — the server-rendered "Need £X.XX in your wallet" message comes through cleanly. Multi-cat discount tier is mirrored client-side from the server's SQL CASE: 1st cat full, 2nd 50% off, 3rd 70% off — fee cards show strikethrough base + discount % chip; invalidating the dashboard query after register/unregister triggers visible price drop on remaining cards.
- **BSL Club-Owner Control + Player Profile + Per-Category Fees**: Club managers (`bslClubs.managerUserId`) get a dedicated control center at `/bsl/my-club` (`client/src/pages/bsl/ClubManager.tsx`) — edit details (name/logo/division/notes), withdraw the club from the league (sets `bslClubs.withdrawnAt` + `isSuspended`; admin-only `POST /api/bsl/clubs/:id/reinstate` clears it), confirm pending join requests (sets `bslPlayers.confirmedByOwnerAt`, fires `bslClubApproved` push), remove players from the roster (clears `bslClubId`/`bslTeamId`/membership rows), and create/delete pairs per category. Pair members are stored in a new join table `bsl_team_members(bslTeamId, bslPlayerId, UNIQUE)` enforcing max 2 per pair, requiring the player to be confirmed AND registered for the team's category, and auto-removing them from sibling pairs in the same category when reassigned. Manager-or-admin authz via helper `loadClubForManager()`. Players get `/bsl/profile` (`client/src/pages/bsl/PlayerProfile.tsx`) — edit `displayName` + `bio` (`PATCH /api/bsl/players/me`) and register/unregister for individual categories (MD/WD/XD) using their BSL wallet balance: `POST /api/bsl/players/me/categories` performs an **atomic conditional UPDATE** with a SQL `CASE` that prices the new category by tier — **1st cat full price, 2nd cat 50% off, 3rd cat 70% off** (`fee*50/100` / `fee*30/100`, integer pence) — keyed off `array_length(categories,1)` so concurrent registrations can never both claim the cheaper tier. The same expression guards `walletBalance >= discountedFee AND NOT (cat = ANY(categories))`. The actual amount charged is derived from the balance delta (`me.walletBalance - updated.walletBalance`) and recorded in a DEBIT `bsl_wallet_transactions` row with the discount tier in the description; `DELETE` removes the cat and any pair memberships (no auto-refund — admin-handled). Admin sets per-category fees in `/bsl/admin/settings` ("Per-category fees" section) — stored in `bslLeagues.categoryFees` jsonb (`{MD,WD,XD}` in pence), defaults `{2500,2500,3000}`, falls back to `playerFee` when blank. New schema columns added via `psql ALTER ADD COLUMN IF NOT EXISTS`: `bslPlayers.{displayName,bio,categories[],confirmedByOwnerAt}`, `bslClubs.withdrawnAt`, `bslLeagues.categoryFees`. Entry points surface in `/bsl` hub: "Manage My Club" (when user is a manager) and "My BSL Profile" (when user is a player).
- **Birmingham Super League (BSL)**: Esports-style competition module at `/bsl`. Locked palette derived from BSL poster (deep navy bg `222 50% 6%`, electric cyan `195 100% 60%`, gold/amber `42 95% 55%`) — no other hues allowed. Public routes: `/bsl` (LeagueMode hub), `/bsl/register-club` (7-step wizard), `/bsl/join` (invite-code flow), `/bsl/wallet`, `/bsl/match/:id`. Admin Control Panel at `/bsl/admin/*` (OWNER/ADMIN, sidebar entry "BSL · Control Panel") with shared `AdminLayout` shell (sticky topbar + animated left nav with framer-motion `layoutId` active bar + pending badge): `/bsl/admin` (Dashboard — 8 stat cards, alert feed, audit log w/ 10s refresh), `/bsl/admin/league` (divisions CRUD, fixture generator, league days), `/bsl/admin/match-day` (drag-drop court grid + rubber status), `/bsl/admin/clubs` (table + flag/suspend/notes editor + invite-code copy + **Mark paid / Mark unpaid** buttons that flip `bslClubs.status` between `ACTIVE` and `PENDING_PAYMENT` via `PATCH /api/bsl/admin/clubs/:id/payment-status` — `PENDING_PAYMENT` hides the club from the public `/api/bsl/clubs` list (filter only returns `ACTIVE` to non-admins); both transitions, plus the legacy `/approve` and `/reject` endpoints, write to `bsl_audit_log` via the `audit()` helper as `club.payment.marked-paid` / `club.payment.marked-pending` / `club.approved` / `club.rejected`), `/bsl/admin/players` (assignment / stats correction / wallet adjust / discipline), `/bsl/admin/payments` (Pending/History tabs + CSV export), `/bsl/admin/media` (upload + tag + MVP/Featured pin), `/bsl/admin/settings` (rules, bank, branding, notifications). Legacy routes `/bsl/admin/verify` and `/bsl/admin/fixtures` alias to the new Payments/Match Day pages. Backend in `server/bsl-routes.ts` with `audit()` helper persisting to `bsl_audit_log`; admin endpoints under `/api/bsl/admin/{dashboard,clubs,players,league-days,transactions,audit,media,pending,payments/export.csv}`. Schema at `shared/schema.ts` (5 enums + 10 tables: bslLeagues/Clubs/Teams/Players/LeagueDays/Fixtures/Rubbers/WalletTransactions/AuditLog/Media; bslLeagues extended with pointsWin/Draw/Loss, matchFormat, courtCount, notificationsEnabled, brandingPrimary/Accent; bslClubs has isFlagged/isSuspended/adminNotes/categories (text[] of MD/WD/XD); bslTeams has category (one team per selected category); bslPlayers has warnings/isSuspended/matchBanCount/disciplineNotes). Payment flow is bank-transfer + uploaded proof to `/uploads/bsl/`, admin-approved. League singleton (`bslLeagues` id=1) seeded with default bank details, £500 club fee, £25 player fee. Components in `client/src/pages/bsl/components/` and admin pages in `client/src/pages/bsl/admin/`. All motion via framer-motion. NOTE: `npm run db:push` may hang on interactive create-vs-rename prompts for new tables — use `psql "$DATABASE_URL"` with `ALTER TABLE … ADD COLUMN IF NOT EXISTS` / `CREATE TABLE IF NOT EXISTS` instead.
- **Universal Notification Engine** (Phase 1): Admin-editable templates + on/off toggle for every event-driven push the app sends. Table `notification_rules` (id, ruleKey UNIQUE, enabled, **category**, title, message, settings jsonb, updatedAt; `category` added via `psql ALTER ADD COLUMN IF NOT EXISTS`). Single source of truth is `RULE_REGISTRY` in `server/notificationRules.ts` — currently ~35 keys grouped by category: Account (accountCreated, accountPasswordChanged, accountRoleChanged), Membership (membershipApproved/Rejected/Expiring/Left), Payments (paymentReceived, paymentRequested, paymentReminder, paymentFailed, paymentRefunded, creditAdded, debtChargeAdded), Sessions (sessionInvited/Booked/Cancelled/Reactivated/Reminder, waitlistPromoted, newSessionMatchingLevel, postSessionUnpaidReminder, attendanceMarked), Rewards (pointsAdded/badgeEarned/redeemed/expiring), League BSL (clubApproved, fixturePublished, matchReminder, resultsSubmitted, walletApproved), Tournaments (published, matchScored), Communication (announcementPosted), Profile (profileIncomplete). `ensureRuleSeeds()` runs on every server boot and **upserts only `category`** so admin-customized title/message/enabled survive code redeploys. Helper `sendRulePush(ruleKey, userIds, vars, opts)` (also exported as `notifyEvent`) fetches the rule (30s in-memory cache, invalidated on PATCH), short-circuits when `enabled=false`, renders `{var}` placeholders, then calls `sendPushToUsers`. Wired trigger sites: `server/auth.ts` register, `server/routes.ts` (session create-invite, cancel, reactivate, payment-PAID, payment-request, payment-reminder, member-left, waitlist-promote, level-match), `server/pushScheduler.ts` hourly unpaid cron, `server/bsl-routes.ts` (club approve, wallet-topup approve). Adding more triggers: append to `RULE_REGISTRY`, restart, then call `sendRulePush(key, userIds, vars, {url, dedupe})` at the trigger site — no DB migration needed. Admin endpoints OWNER/ADMIN-only: `GET /api/admin/notification-rules`, `PATCH /api/admin/notification-rules/:key` (key allowlisted against `RULE_KEYS`, blank title/message rejected). UI at `/admin/notification-rules` (`client/src/pages/admin/NotificationRules.tsx`) — rules grouped into collapsible category sections (Account / Membership / Sessions / Payments / Rewards / League (BSL) / Tournaments / Communication / Profile / General) with on-count badge per section, one card per rule with Switch (auto-saves on toggle), Title/Message editors, dynamic `{placeholder}` chips extracted from the live template, Save button. Sidebar entry "Auto Reminders" in admin group.
- **Smart Missing-Info Targeting (Phase 2)**: Weekly cron `runProfileIncompleteReminder` in `server/notificationCrons.ts` (wired in `server/index.ts`, runs once 60s after boot then every 7 days) scans `users.phone IS NULL` and `playerProfiles.gender IS NULL` and fires the `profileIncomplete` rule with `{field}` filled (push + in-app), deduped per user/field via `push_send_log` (`refType: "profile-missing-{field}"`). Adding new field nags = append a scan block to `runProfileIncompleteReminder` with a unique `dedupe.refType`.
- **Scheduled Broadcasts + Test/Preview (Phase 4)**: Table `notification_schedules` (id, ruleKey?, title?, message?, url?, segment jsonb, vars jsonb, scheduleAt, status pending|sending|sent|cancelled|failed, createdBy, sentAt, errorMessage). Cron `runScheduledNotifications` sweeps every 60s, atomically claims rows (status `pending` → `sending`), resolves segment to userIds (USER/CLUB/TEAM/TOURNAMENT/ALL), then dispatches via `sendRulePush` (rule-driven) or `sendPushBySegment` (ad-hoc). Endpoints (OWNER/ADMIN; ALL segment OWNER-only): `GET/POST /api/admin/notification-schedules`, `DELETE /api/admin/notification-schedules/:id` (cancels pending only). Admin Push Broadcast page (`/admin/push-broadcast`) has a `datetime-local` "Schedule for later" input — when set, the **Send now** button disables and a **Schedule** button appears, plus an "Upcoming & recent" list with status badges and one-click cancel for pending rows. Per-rule **Test send to me** (`POST /api/admin/notification-rules/:key/test` with optional `vars`) lives on each rule card in `/admin/notification-rules`, alongside live placeholder inputs that drive a client-side **Preview** card showing the rendered title/message.
- **Notification Analytics + Email Channel (Phase 5)**: Table `notification_send_metrics` (id, ruleKey?, channel push|inapp|email, recipientsCount, sentAt) is appended once per channel dispatch inside `sendRulePush` (after channel-pref filtering). Endpoint `GET /api/admin/notification-rules/:key/stats?days=30` aggregates SUM(recipientsCount) and COUNT(*) per channel; surfaced in the rule card via a collapsible **Stats** panel (3 tiles: Push / In-app / Email). Email channel implemented in `server/emailSender.ts` using Resend REST API (`https://api.resend.com/emails`, secret `RESEND_API_KEY`, `EMAIL_FROM` env defaults to "Club Master <notifications@clubmaster.app>"). `sendEmailToUsers(userIds, {subject, html, text?}, ruleCategory)` filters via `getOptedInUserIdsByCategory(channel="email")`, joins `users.email`, and sends a single multi-recipient email (max 50/batch). `sendRulePush` accepts `opts.email: true` to enable email per call site (still no auto-email on existing trigger sites — opt-in only). `wrapEmailHtml(title, message, url?)` provides a minimal styled wrapper.
- **User Notification Preferences (Phase 3)**: Per-user category × channel matrix backed by `user_notification_prefs.categoryPrefs` JSONB column (added via `psql ALTER ADD COLUMN IF NOT EXISTS`). Shape: `{ "<Category>": { push?: bool, inapp?: bool, email?: bool } }` — missing entries default to opted-in. Categories mirror RULE_REGISTRY (Account / Membership / Sessions / Payments / Rewards / League (BSL) / Tournaments / Communication / Profile / General). Backend filter `getOptedInUserIdsByCategory(userIds, ruleCategory, legacyKey, channel)` in `server/oneSignal.ts` honors the JSONB matrix AND legacy boolean columns (back-compat with the old 5 flags). `sendPushToUsers` accepts an optional `ruleCategory` arg; when supplied, push channel filtering switches to the matrix. `sendRulePush` always passes the rule's category and accepts `opts.inApp: true` to additionally write a row into `notifications` (filtered by the inapp channel pref). UI at `/settings/notifications` (`client/src/pages/NotificationSettings.tsx`) renders a category × channel grid with optimistic-update toggles + a per-row "All" master toggle; categories are fetched from `GET /api/notifications/categories` (DISTINCT on notification_rules). PATCH `/api/notifications/preferences` accepts either legacy flags or `categoryPrefs` partial patches and shallow-merges per category.
- **Push Notifications (OneSignal)**: Web Push via OneSignal v16 SDK (script in `client/index.html`, worker at `client/public/OneSignalSDKWorker.js`). Init + external-id login + player-id registration handled by `client/src/components/OneSignalBootstrap.tsx` (mounted in `App.tsx`) using helpers in `client/src/lib/oneSignal.ts`. Backend: `server/oneSignal.ts` (REST helper, `Key` auth on `https://api.onesignal.com/notifications`, `include_aliases.external_id`); `server/notificationRoutes.ts` (POST `/api/notifications/register`, GET/PATCH `/api/notifications/preferences`, OWNER/ADMIN POST `/api/admin/notifications/send` with segments USER/CLUB/TEAM/TOURNAMENT/ALL — ALL is OWNER-only; preference `adminAnnouncement` filters broadcast recipients). Trigger hooks in `server/routes.ts`: payment-override PAID → `paymentReceived`, promote-waiting → `waitlistPromoted`, session create → `newSessionMatchingLevel` for club members whose `playerProfiles.grade` falls within `skillLevelMin/Max` (uses `GRADE_ORDER`). Hourly cron `runPostSessionUnpaidReminder` in `server/pushScheduler.ts` (wired in `server/index.ts`) fires `postSessionUnpaidReminder` once per signup (deduped via `push_send_log`) for finished sessions in the last 7 days where `paymentStatus=UNPAID`. Tables (created via `psql ALTER`, mirrored in `shared/schema.ts`): `user_push_subscriptions`, `user_notification_prefs`, `push_send_log`. Routes: `/settings/notifications` (user prefs + enable push), `/admin/push-broadcast` (admin compose + segment send). Sidebar: "Push Settings" in comms group, "Push Broadcast" in admin group. Secrets: `VITE_ONESIGNAL_APP_ID`, `ONESIGNAL_REST_API_KEY`.
- **Player Rankings UI** (`client/src/pages/PlayerRankings.tsx`): Dribbble-style rebuild — top 3 rendered as `HeroPlayerCard` (image-bleed gender-aware hero photo from `attached_assets/hero_{male,female}_player.png`, violet glow ring on avatar, overlaid 3-up stat strip Win%/W-L/Pts, win-index gradient progress bar, pill action chips, place chip top-left, kebab top-right). Ranks 4+ render as `CompactPlayerCard` (glassy row with rank pill, avatar, heat bar, W/L + Pts strip). Filter bar uses violet `FilterPill` for scope toggle. All surfaces use `hsl(var(--card)/x)` tokens — no hardcoded slate colors.

## Product

- **Club Management**: Multi-club support, membership management, recurring events, session scheduling, and financial intelligence.
- **Player Engagement**: Dynamic player ranking, match organization, individual player profiles, achievement badges, and AI-powered analytics.
- **Admin Tools**: Comprehensive user, club, venue, and administrator management, audit logs, helpdesk ticketing, and an OWNER-only Club Control Center for feature toggling.
- **Advanced Features**: AI-powered reporting, session schedule generation, 3D avatar selection, league management, tournaments, merchandise system, and a community hub.
- **Monetization**: Freemium model, tiered session fees, credit request system, and club-scoped referral programs.

## Gotchas

- Express JSON body limit is path-aware in `server/index.ts`: default 256kb for the API surface, with `/api/bsl/clubs*` opted into an 8mb cap so the BSL club registration flow can accept the base64 image fallback when the file-upload endpoint is unavailable. Raise the cap on additional routes by extending `jsonLargeRoutes`, never globally.
- Legacy hardcoded color blocks in some UI components (e.g., `Dashboard.tsx`, `Deals.tsx`, `MerchandisePage.tsx`, `not-found.tsx`) may bypass theme tokens and not harmonize with selected themes. These require refactoring to use CSS variables.
- Merchandise Supplier Order Sheet explicitly avoids selecting or displaying any customer contact details (email, phone) for privacy.
